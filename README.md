## 🧠 Project Overview

**Rupee Roast** is a full-stack AI finance app that transforms raw expense data into sarcastic, data-backed commentary ("roasts") designed to make financial awareness engaging and memorable.

Instead of dull monthly reports, Rupee Roast gives users:
- brutally honest spending analysis,
- visual trend breakdowns,
- category-level diagnostics,
- and practical recommendations generated by **Google Gemini**.

At its core, the project balances **personality + performance**:
- a playful neo-brutalist UI on the frontend,
- robust analytics and caching on the backend,
- and AI generation pipelines tuned for speed, cost control, and reliability.

If your wallet had a stand-up comedian and a financial advisor in one room, this would be the result.

---

## 🎬 Demo Description

Rupee Roast's user journey is designed to feel fast, entertaining, and useful:

1. User logs in securely via JWT-authenticated flow.
2. User submits or updates expenses.
3. Dashboard instantly visualizes spending by category and timeline.
4. Backend checks whether expense data changed using hashing logic.
5. If unchanged, a cached roast is returned in milliseconds (~2ms typical cache hit).
6. If changed, Gemini generates a fresh roast with data context.
7. The new roast is cached in MongoDB for future requests.
8. UI displays roast + insights with charts and animated cards.

This makes repeated usage highly responsive while preserving AI quality when data changes.

## 🏗️ System Architecture

Rupee Roast follows a clean full-stack separation with a service-oriented backend.

```text
Client (React + Vite)
  ├─ Auth UI / Dashboard UI / Charts / Roast Cards
  └─ API calls (JWT-protected)
          │
          ▼
Server (Express)
  ├─ Routes
  ├─ Controllers (business flow + expense hash checks)
  ├─ Services (Gemini calls, cache lookup/store, rate limiting)
  └─ Models (MongoDB persistence)
          │
          ▼
MongoDB
  ├─ User/Expense data
  └─ Roast cache documents (prompt/hash → response)
          │
          ▼
Google Gemini API
  └─ Fresh roast generation when cache miss or data changed
```

### Core Architecture Roles

- **`server/services/geminiService.js`**  
  Handles AI prompt execution, response caching, and rate-limiting guardrails.
- **`server/models/Roast.js`**  
  Stores generated roast payloads keyed for fast retrieval.
- **`server/controllers/*`**  
  Uses hashing logic to detect whether a user's expenses changed before deciding cache hit vs regeneration.

## 🔐 Environment Variables Setup

Create a `.env` file in the `server/` directory:

```env
PORT=5000
MONGO_URI=your_mongodb_connection_string
JWT_SECRET=your_super_secret_jwt_key
GEMINI_API_KEY=your_google_gemini_api_key
```

### Variable Reference

| Variable | Required | Description |
|---|---|---|
| `PORT` | Yes | Backend server port |
| `MONGO_URI` | Yes | MongoDB connection URI |
| `JWT_SECRET` | Yes | Secret used to sign/verify JWT tokens |
| `GEMINI_API_KEY` | Yes | API key for Gemini model access |

> Security note: Never commit `.env` files to version control.

## 🧪 AI Pipeline Explanation

Rupee Roast's AI system is structured for both quality and efficiency.

1. **Data Collection**  
   User expense data is aggregated into a structured financial context.
2. **Change Detection**  
   Controllers compute/check a deterministic hash of relevant expense state.
3. **Cache Lookup**  
   Existing roast entry is searched in `Roast` collection by user + hash/signature.
4. **Rate-Limit Gate**  
   Outbound Gemini calls pass through backend rate-limiting logic.
5. **Prompt Construction**  
   Prompt template injects spending insights and desired roast tone constraints.
6. **Gemini Generation**  
   `gemini-1.5-flash-latest` returns roast + guidance-style narrative.
7. **Cache Persist**  
   Fresh response is stored for future hits.
8. **Client Delivery**  
   Response returns to dashboard with near-instant subsequent retrieval.

## 🗃️ Caching Strategy

Caching is one of Rupee Roast's core differentiators.

### Why it exists
- AI calls are relatively expensive and quota-bound.
- Many users repeatedly request analysis without changing expenses.
- Re-generation is unnecessary when financial state is unchanged.

### How it works
- Input-dependent hash identifies a unique expense state.
- If hash matches prior roast entry, return cached payload.
- If hash differs, generate new roast and update cache.

### Benefits
- **Speed:** cache hits around ~2ms.
- **Cost efficiency:** fewer Gemini API calls.
- **Reliability:** fewer external dependencies for repeat requests.
- **Determinism:** same state yields same roast version unless regenerated by design.

---

## 🚦 Rate Limiting Strategy

Rupee Roast includes API-side controls to avoid model quota failures and unstable bursts.

### Goals
- Prevent Gemini overuse from rapid user retries.
- Reduce risk of hard failures during traffic spikes.
- Keep UX responsive and predictable.

### Practical behavior
- Gate AI generation requests before external API call.
- Enforce request pacing for roast generation paths.
- Prefer cache retrieval whenever possible to bypass quota spend.

### Outcome
A stable AI layer that stays performant under normal usage while respecting provider constraints.

## ⚙️ Performance Optimizations

- **Hash-based invalidation** avoids unnecessary model calls.
- **Mongo-backed roast cache** cuts end-to-end response latency.
- **Separation of concerns** in controllers/services reduces overhead and complexity.
- **Vite + React 19** improves frontend startup and update performance.
- **Lean payload shaping** for dashboard responses keeps UI fast.
- **Rate-limited AI path** protects against cascading timeout/quota issues.
